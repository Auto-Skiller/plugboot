# Engine Architecture & Context Control Audit

## 1. Directory & File Overview
- `.engines/` contains only `.context_control.engine/` directory which in turn holds three core engine protocols (`cataloger.engine.md`, `navigator.engine.md`, `router.engine.md`).
- `.brain/.engines.context_control/` contains `core_engines.catalog.yaml`, `engines.rules.yaml`, and `extended_engines.catalog.yaml`.
- `core_engines.catalog.yaml` successfully references the three engines in `.engines/.context_control.engine/`.
- `extended_engines.catalog.yaml` is empty, serving as a placeholder for future domain-specific engines.
- `.brain/.catalogs.index.yaml` serves as the master routing map.

## 2. Gap Analysis
- The paths referenced in `core_engines.catalog.yaml` are slightly off compared to actual file structure. The files are located in `.engines/.context_control.engine/` (note the dot in `.context_control.engine`), but the catalog references them as `.engines/context_control.engine/` (missing the dot). This will break automated path resolution.
- The `.engines` directory is currently sparse, containing only the context control engines.
- The `engines.rules.yaml` mandates semantic versioning, engine IDs, and last updated dates in required frontmatter for `.engine.md` files.

## 3. Permanent Solutions
- **Fix Catalog Paths:** Update `.brain/.engines.context_control/core_engines.catalog.yaml` to include the leading dot for `.context_control.engine/` in all `path` entries.
- **Implement Frontmatter:** Update `cataloger.engine.md`, `navigator.engine.md`, and `router.engine.md` to include the required frontmatter specified in `engines.rules.yaml` (engine_id, version, last_updated, cataloger_lock).
- **Enforce Syncing:** The `cataloger.engine.md` should be strictly followed to ensure catalog updates correctly sync with actual file paths and metadata.
