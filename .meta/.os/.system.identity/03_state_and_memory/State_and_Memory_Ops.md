# 💾 State and Memory Ops

## 1. DB Navigation & Responsibilities (`.db/`)
- **The Map Principle:** `.db/` tells you **WHERE** things are. `.db/.system.board.yaml` is the routing root.
- **Path Integrity:** Never guess file paths. Use the absolute paths provided in DB files or `.db/.system.board.yaml`.

### 1.1 DB File Responsibilities
The OS state is driven by a series of physical `.yaml` databases inside `.db/`. These are the ultimate source of truth:
- **`.db/.system.board.yaml`**: The master board file. It holds the global `work_mode`, system-wide queues (like the evolution queue), and global telemetry.
- **`.db/pipeline_*.board.yaml`**: The state for specific continuous workflows (Scaler, Hustler). They track pipeline-specific modes, incoming data queues, and local health metrics.
- **`.db/project_*.board.yaml`**: The index for finite codebases. It tracks project status and completion metrics.
- **`.db/toolboxes.board.yaml`**: The capability registry. It tracks all available agent skills and cross-toolbox dependencies.

## 2. The Schema Contract (`.db/.schemas/`)
Every DB file is strictly governed by a YAML schema. Schemas enforce the structural contract between agents and the sync daemon.
- **Strict Compliance:** Agents must NEVER invent new fields, status strings, or array structures. Before writing to a DB file or a milestone session, consult the corresponding schema to ensure the data shape is valid.
- **Mapping:** The mapping of which schema controls which DB file is explicitly listed in `.db/.system.board.yaml` under the `db_index` block.

## 3. Reading the Controller
`.db/.system.board.yaml` is the Central Command State — the single top-level dashboard for the entire OS.
- **Comment Markers are Law:** The `# from [source]` comments on every block define the sync direction. NEVER remove or alter them.
- **Engine-Owned Rollups:** All blocks marked `# from [source]` are written by the sync daemon from their respective source DB. Agents should NOT hand-edit them directly — edit the source DB instead.

## 4. The Dual-Entry System
Every field in a YAML file carries an entry type, indicated by an inline comment:
- **`# (in)`** — **Top-Down (high-to-low).** This field is set by a higher-level file (user or parent DB). Lower-level files must respect it and must not override it.
- **`# (in-out)`** — **Bidirectional.** Both agents and the daemon can read and write this field. Used for hubs, queues, and message passing.
- **No comment (OUT)** — **Bottom-Up.** Generated by child files or the daemon. The daemon aggregates these UP to the parent DB (and ultimately to `.db/.system.board.yaml`).

The sync daemon uses these markers to determine sync direction: it pushes `(in)` fields downward and aggregates bottom-up data upward.

## 5. Freshness Model
Since the sync daemon is **event-driven** (it watches for file changes rather than polling on a timer), freshness is tracked with just two fields in `system_metadata`:
```yaml
freshness:
  last_synced: timestamp   # audit trail — when the daemon last processed this file
  status: fresh            # fresh | syncing | error
```
- `syncing` — daemon sets this before writing to the file (acts as a soft lock signal).
- `error` — daemon sets this if a write fails, so agents know the state is unreliable.
- There is no `fresh_until` or `threshold_seconds` — the daemon handles staleness detection via file-watch events, not TTL.

## 6. Sessions as Mid-Level Boards
While overarching milestone `status` is a Top-Down state in `.db/.system.board.yaml`, the execution details (goal `status`, goal `priority`, and `max_rounds`) are Top-Down states **localized** to the `SESSION.yaml` file. The user edits these directly in `SESSION.yaml`, making the session a mid-level dashboard for its specific task.

## 7. The Sync Daemon
The sync daemon is an always-running event-driven process that:
1. **Watches** all DB files for changes.
2. **On change:** determines sync direction using `# (in)` / `# (in-out)` markers.
3. **Propagates** changes up or down the hierarchy accordingly.
4. **Updates** `freshness.last_synced` and `freshness.status` on each processed file.

Agents do NOT need to manually trigger sync. Any structural action (creating a folder, finishing a task, mutating state) is automatically detected and propagated by the daemon.

## 8. Sync & Health Lifecycle Events
Events emitted by the daemon regarding state sync:

| Event | Emitted by | Payload (minimum) | Consumer |
|---|---|---|---|
| `SYNC_STARTED` | `.infra/engine.py` boot | `triggered_by`, `at` | `hub.recent_events` |
| `SYNC_COMPLETED` | `.infra/engine.py` exit | `started_at`, `completed_at`, `outcome` | `hub.recent_events` |
| `SYNC_FAILED` | `.infra/engine.py` exception | `started_at`, `failed_at`, `error` | `hub.messages` (severity: ERROR) |
